# Выводы по проекту refine_llm

## Что сделали

Построили end-to-end пайплайн для подготовки fine-tuning классификатора запросов разработчика к AI-ассистенту по кодовой базе. Решили классификацию на 4 класса:

| Класс | Что значит |
|---|---|
| `search` | Найти/локализовать код (где X, найди вызовы Y) |
| `understand` | Объяснить конкретный код или функцию |
| `describe` | Высокоуровневое описание модуля/архитектуры |
| `modify` | Изменить код: рефакторинг, генерация, фикс, тест |

Готовы три датасета, валидация, сплит, baseline, клиент для запуска ФТ. Запуск ФТ отложен по запросу пользователя.

## Три датасета и зачем их три

| Датасет | Размер | Eval | Назначение |
|---|---|---|---|
| `data/` (generic) | 50 | 10 | Sanity-check пайплайна. Нейтральные RU IT-промпты. |
| `data/zeal/` | 50 | 10 | Доменно-специфичный. Промпты про реальный проект Zeal (macOS disassembler), используют Mach-O / cfg.c / AINamingService. |
| `data/hard/` | 200 | 40 | Adversarial bilingual. RU + EN + code-switched. Сложные неоднозначные промпты, отрицания, гибридные интенции. |

Зачем три: каждый показывает разный уровень сложности задачи и разную сторону поведения модели. Generic — легко, zeal — средне, hard — реалистично.

## Baseline-результаты (`gpt-4o-mini` без ФТ)

| Датасет | Eval size | Accuracy | Errors | Format strict | Classes predicted |
|---|---|---|---|---|---|
| `data/` (generic) | 10 | **90%** | 1 | 100% | 4/4 |
| `data/zeal/` | 10 | **80%** | 2 | 100% | 4/4 |
| `data/hard/` | 40 | **85%** | 6 | 100% | 4/4 |

Per-class на `data/hard/` (40 eval, по 10 на класс):

| Класс | Accuracy |
|---|---|
| search | 70% (7/10) |
| understand | 100% (10/10) |
| describe | 70% (7/10) |
| modify | 100% (10/10) |

## Главные выводы

### 1. Baseline GPT-4o-mini сильнее, чем казалось
На простых датасетах модель достигает 80-90% без ФТ. Прирост от ФТ на чистых данных будет маленьким (1-2 п.п.). Чтобы продемонстрировать осмысленный прирост, нужен сложный, неоднозначный датасет.

### 2. Модель отлично различает явные интенции, плохо — гибриды
- `understand` и `modify` распознаются почти идеально, когда есть явные глаголы-маркеры (`почему`, `объясни`, `добавь`, `fix`).
- Путаница начинается когда:
  - **describe ↔ understand**: вопрос про "состояние проекта" / "что дальше" — модель тянет в understand
  - **search ↔ understand**: запрос статуса без явного "где" ("check whats next") — тоже идёт в understand
  - **search ↔ modify**: фразы с двойным глаголом ("check ... and copy") — модель цепляется за второй глагол

### 3. Code-switched промпты не сильно ломают baseline
Смешанные RU+EN промпты (`refactor функцию validateConfig`, `где parser initializes?`) gpt-4o-mini обрабатывает уверенно. Падение accuracy на bilingual датасете объясняется не языком, а намеренной adversarial-разметкой.

### 4. Идеальный формат — единственное "узкое место" с 100% результатом
На всех трёх датасетах **format strict OK = 100%**. Модель никогда не возвращает мусор вокруг ярлыка (нет `"answer: search"` или `search.\n` etc.) при правильно сформулированном system prompt, требующем "ровно одно слово". Это значит, что метрика "формат" в criteria.md уже на потолке — ФТ не сможет её улучшить.

### 5. Размер датасета имеет значение для статистической значимости
- 10 eval → 1 ошибка = 10% accuracy. Шум огромный.
- 40 eval → 1 ошибка = 2.5%. Различия гораздо надёжнее.
- Для значимой per-class accuracy нужно минимум 5-10 примеров на класс в eval, т.е. датасет от 100 примеров. Hard-датасет (200) — комфортный размер.

### 6. Реальные данные доступны прямо в `~/.claude/projects/`
Самый ценный источник — собственные истории Claude Code сессий пользователя. Из них агент извлёк 120+ кандидатов за 4 минуты. Это эффективнее любого парсинга интернета:
- Реальное распределение интенций (без cherry-picking)
- Натуральный язык (грубоватый, с опечатками, code-switched)
- Уже отфильтровано по теме "запросы к AI о коде"

### 7. Перекос real-данных по классам предсказуем
В реальных Claude Code сессиях `modify` доминирует (~38%), `search` редкий (~16%). Без балансировки синтетикой получится перекошенный датасет. Hard-датасет вышел сбалансированным (50/50/50/50) только за счёт целевой генерации недостающих классов.

### 8. Стратегия снижения baseline для демо ФТ работает
Adversarial-промпты (отрицания, гибриды интенций, без явных глаголов) реально снижают baseline:
- 90% (generic) → 80% (доменный) → 85% (bilingual+adversarial на 4× большем eval)
- Видимых ошибок: 1 → 2 → 6
- 6 ошибок — заметная цель для ФТ, можно показать осмысленный прирост

## Чего достичь сложнее

### Опустить baseline ниже 75%
Требует более тонкой adversarial-разметки:
- `describe`-промпты без слов "архитектура / опиши / модули"
- `search`-промпты с action-глаголами ("проверь и скажи где...")
- `modify` без явных команд ("давай чтобы Х было через Y")
- Sarcastic/rhetorical тон ("серьёзно, это работает?")

### Метрика "формат" уже на 100%
Для строгой задачи (одно слово из закрытого списка) ФТ не даст прирост по этой оси. Если хотим разнообразных метрик, нужно либо усложнять формат (структурированный JSON-ответ), либо менять задачу на open-ended (объяснение, генерация).

## Что осталось для запуска ФТ

Код готов, файлы готовы. Нужно только:

```bash
.venv/bin/python scripts/finetune_client.py --go --data-dir data/hard
# ждать ~5-15 мин, ~$0.30
.venv/bin/python scripts/baseline.py --data-dir data/hard \
    --model ft:gpt-4o-mini:... --output results/finetuned_hard.json
```

После прогона сравнить `baseline_hard.json` vs `finetuned_hard.json` по 5 критериям из `results/criteria.md`.
